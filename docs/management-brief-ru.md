# Краткий технический бриф для руководства

## Что решает система

Система обеспечивает управляемое обратное подключение удалённых агентов к
центральному `reverse_ssh`, не требуя открывать входящий SSH-порт на стороне
агента. Агент сам устанавливает исходящее соединение с публичной VPS через
обычный HTTPS/WSS-порт `443`, после чего оператор может подключиться к агенту
через центральную catcher-консоль.

Дополнительный контур мониторинга фиксирует подключения, определяет реальный
источник и VPS-маршрут, показывает активные сессии в Dashboard и отправляет
Telegram-уведомления.

## Схема в одном экране

```text
Удалённый агент
    |
    | исходящий WSS/HTTPS :443
    v
Публичная VPS: nginx + TLS + decoy-сайт
    |                         |
    | proxy transport        | mirror метаданных подключения
    v                         v
Центральный reverse_ssh     nginx-edge-forwarder
    |                         |
    | lifecycle webhook       | ingress event
    +-------------+-----------+
                  v
             rssh-logger
        SQLite / Dashboard / Telegram
                  ^
                  |
        session reconciler (`reverse_ssh ls`)
```

## Что такое обратный туннель

При обычном SSH оператор инициирует соединение к удалённой машине, поэтому на
ней должен быть доступен входящий порт. В обратной схеме соединение инициирует
сам агент наружу. Это удобно для машин за NAT, корпоративным firewall или
динамическим адресом.

После установления исходящего канала центральный сервер связывает логический ID
агента с уже открытым транспортом. Оператор подключается к этому ID через
catcher-консоль; новое прямое соединение к удалённой машине не требуется.

Ключевой эффект: удалённый агент не публикует SSH в интернет, а центральная
точка управления контролирует доступ и ведёт аудит сессий.

## Зачем нужна публичная VPS

VPS выполняет роль edge-входа:

- принимает стандартный `443/tcp`, который обычно разрешён для исходящего
  трафика;
- завершает TLS и обслуживает обычный decoy-сайт на посторонних URL;
- проксирует только известные случайные WSS/HTTPS transport paths;
- перенаправляет транспорт на центральный `reverse_ssh`;
- независимо отправляет метаданные входа в центральный logger;
- позволяет менять публичный адрес или домен без переноса центрального
  хранилища и панели мониторинга.

Несколько VPS дают разные публичные точки входа и уменьшают зависимость от
одного адреса или провайдера. Центральный сервер остаётся единой точкой
управления, корреляции и хранения.

## Интересные детали проксирования

### WSS и HTTPS polling

Поддерживаются два веб-транспорта:

- WSS — длительное WebSocket-соединение;
- HTTPS polling — начальный запрос и последующий обмен обычными HTTP-запросами.

Снаружи оба варианта выглядят как HTTPS-трафик к VPS. Внутри полезная нагрузка
содержит собственную SSH-аутентификацию `reverse_ssh`; TLS nginx не заменяет
проверку ключей внутри туннеля.

### Nginx mirror

Nginx одновременно проксирует рабочий запрос и создаёт внутренний mirror-запрос
с метаданными: адрес клиента, VPS, transport path, время и заголовки. Mirror не
находится в основном канале данных и не должен задерживать агентское
соединение. Локальный forwarder временно складывает события в spool и
доставляет их на центральный logger.

### Два разных адреса logger

Контейнеры на центральном сервере используют Docker DNS:

```text
http://rssh-logger:8080
```

VPS используют опубликованный и ограниченный firewall адрес:

```text
http://<main-bind-ip>:8080
```

HTTP в этом примере допустим только внутри изолированного шифрованного канала
(например, VPN). Если маршрут проходит через публичную или недоверенную сеть,
нужен HTTPS с проверкой CA, предпочтительно mTLS: firewall и токен не шифруют
содержимое запроса.

Эти пути нельзя смешивать. Внутренний lifecycle webhook через публичный IP
может не пройти из-за firewall или hairpin routing, даже если VPS успешно
доставляют ingress на тот же внешний адрес.

## Откуда берётся информация о подключении

Система использует три независимых сигнала:

1. `reverse_ssh` webhook — каноническое событие `connected/disconnected`, ID,
   hostname и данные, которые видит центральный listener.
2. Nginx ingress — реальный внешний клиентский IP, конкретная VPS, домен,
   transport и время входа.
3. Session reconciler — периодический `reverse_ssh ls`, то есть фактический
   список живых ID на текущий момент.

Logger сопоставляет webhook и ingress по IP-признакам, trusted-proxy данным и
временным окнам. При успешном сопоставлении Dashboard и Telegram получают
обогащённое событие: кто подключился, через какую VPS и с какого адреса.

Reconciler нужен как контроль состояния. Если webhook потерян, активная сессия
всё равно будет видна, но без описательных полей. Это диагностический сигнал,
а не полноценное lifecycle-событие.

## Что произошло при последней диагностике

Наблюдалась следующая комбинация:

- ingress от VPS поступал;
- reconciler видел живого агента;
- свежего lifecycle webhook в БД не было;
- поэтому matcher не имел события, к которому можно привязать ingress;
- Dashboard показывал синтетическую `reconciled` строку без client/network;
- Telegram не отправлялся, потому что он запускается lifecycle webhook-событием.

Webhook был зарегистрирован на публичный адрес центрального logger. Из
контейнера `reverse_ssh` этот адрес не работал как надёжный внутренний маршрут.
После регистрации через `http://rssh-logger:8080` события, корреляция и
Telegram-уведомления восстановились.

Практический вывод: наличие живой сессии ещё не доказывает работоспособность
webhook. Нужно контролировать согласованность всех трёх сигналов.

## Безопасность

- Удалённые агенты инициируют соединения наружу; входящий SSH на них не
  публикуется.
- Публичные VPS открывают только необходимые `80/443` и ограниченный
  административный SSH.
- Центральные `reverse_ssh` и logger доступны только allowlist адресам VPS и
  операторов.
- Dashboard не публикуется через публичный decoy nginx; предпочтителен SSH
  tunnel.
- Lifecycle webhook остаётся внутри Docker network.
- Webhook, ingress, health, dashboard и Telegram используют разные токены или
  ключи с разными областями доступа.
- Для Docker-портов проверяется не только UFW, но и `DOCKER-USER`.
- Реальные IP, hostname, ключи и токены не хранятся в репозитории.
- Раскрытый URL-token подлежит немедленной ротации.

## Наблюдаемость и аудит

Центральный logger хранит:

- сырые lifecycle webhook-события;
- сырые ingress-события;
- обогащённые результаты корреляции;
- session snapshots;
- ошибки подключения `reverse_ssh`;
- статусы доставки Telegram;
- состояние edge-health для каждой VPS.

Данные сохраняются в SQLite и append-only JSONL. Dashboard показывает активные
и недавние сессии, корреляцию, состояние VPS и системные события.

## Вероятные вопросы и короткие ответы

### Почему нельзя оставить один прямой сервер?

VPS отделяет публичный edge от центральной точки управления, позволяет
использовать обычный HTTPS/WSS, менять адреса входа и ограничивать доступ к
центральному серверу allowlist-правилами.

### Почему используется порт 443?

Агент создаёт исходящее веб-соединение. `443/tcp` наиболее совместим с NAT,
корпоративными firewall и системными proxy.

### Можно ли получить доступ к произвольной машине?

Нет. Нужен заранее сгенерированный агент с корректными ключами и fingerprint.
Оператор также проходит SSH-аутентификацию на центральной catcher-консоли.

### Видна ли реальная точка подключения?

Да, nginx ingress фиксирует внешний IP и VPS-маршрут. Эти данные
сопоставляются с lifecycle webhook.

### Что будет при потере Telegram?

События остаются в SQLite/JSONL. Telegram — канал оповещения, а не источник
истины. Статус каждой попытки доставки сохраняется отдельно.

### Что будет при перезапуске контейнеров?

Данные находятся в persistent volume. После замены `reverse_ssh` необходимо
проверить runtime-регистрацию webhook. Compose запускается с полным набором
base + edge override, иначе публикация logger может исчезнуть.

### Где основные точки отказа?

- публичная VPS или её домен;
- маршрут VPS → central;
- центральный `reverse_ssh`;
- runtime webhook registration;
- центральное хранилище.

Несколько VPS снижают edge-риск, session reconciler обнаруживает расхождение
событий и реального состояния, а persistent storage сохраняет аудит.

## Ближайшие улучшения

1. Автоматически проверять регистрацию lifecycle webhook после recreate.
2. Создать alert на состояние «ingress или live snapshot есть, webhook нет».
3. Автоматизировать синхронизацию VPS allowlist в UFW и `DOCKER-USER`.
4. Регулярно проверять восстановление SQLite/JSONL из резервной копии.
5. Ввести плановую ротацию ключей и токенов с журналом изменений.
6. Зафиксировать SLO: доступность edge, задержка webhook и задержка Telegram.

## Формат короткого доклада

Для устного доклада на 5–7 минут достаточно:

1. Задача: безопасный доступ к агентам за NAT без входящих портов.
2. Путь: агент → VPS nginx `443` → central `reverse_ssh`.
3. Наблюдаемость: webhook + ingress + reconciler.
4. Результат: Dashboard, аудит и Telegram.
5. Безопасность: ключи, allowlist, private Docker webhook, разделённые токены.
6. Последний кейс: live-сессия была, webhook шёл на неверный public route;
   Docker DNS восстановил полный event flow.
7. Следующий шаг: автоматический контроль расхождения трёх сигналов.
